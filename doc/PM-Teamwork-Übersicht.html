Das Problem mit geteilten Tools

Jeder hat seinen eigenen Claude-Kontext.

Niemand weiß was der andere gerade macht.

Skills die einer verbessert, bleiben bei ihm.

Learnings die einer macht, gehen verloren.

Das System löst das — aber nur wenn ihr wisst wie.

Das Grundprinzip

Dein Zimmer

Sandbox. Dein Denkraum.

Niemand sieht rein.

Kein Format-Zwang.

output/YYYY-MM-DD/

scripts/

Drafts & Prototypen

Das Wohnzimmer

Truth. Gilt für alle.

Familienregeln.

Nur per Review rein.

CLAUDE.md

context/

.claude/skills/

Der Übergang wird bewacht — nicht automatisch, nicht heimlich.

Die drei Schichten

SchichtWasWer richtet ein
1 — Geteilte Infrastruktur Git Repo: CLAUDE.md, context/, .claude/skills/ Einmal, gemeinsam
2 — Geteilte Quellen MCPs: Slack, Granola, Linear, Notion... Jede Person selbst
3 — Persönlicher Workspace Lokal: output/, scripts/, Experimente Jede Person selbst

Geteilte Quelle ≠ geteilte Interpretation. Ein PM und ein Designer lesen denselben Slack-Thread — und sehen verschiedene Dinge. Das ist gut so.

Skills-Verbesserungen gehen per PR ins Repo. Learnings in context/learnings.md.

Learnings schließen die Schleife

Ein Tool lernt nicht.

Ein System das durch Benutzung besser wird, tut es —

— aber nur wenn Menschen die Schleife schließen.

"FAQ-Pattern war klar besser als Chat-Interface. User wollen suchen, nicht chatten."
→ context/learnings.md → PR → alle Skills wissen es beim nächsten Mal.

Das ist der schwierigste Schritt. Nach dem Handoff hat niemand mehr Energie. Deshalb: expliziter Schritt, kein Nachgedanke.

Was ihr als Team festlegen müsst

FrageWarum es zählt
Wer ist der Guardian des Wohnzimmers? Wer merged PRs, hält context/ aktuell, hat Skill-Überblick
Wie läuft ein Skill-PR ab? Wer reviewed, wie schnell, welche Kriterien
Wann ist ein Artefakt "geteilt genug"? Draft-Brief im Slack vs. finaler Brief im Meeting
Wie fließen Learnings zurück? Nach jedem Flow? Wöchentlich? Wer schreibt sie?
Wer onboardet neue Team-Mitglieder? Repo erklären, MCPs einrichten, erster gemeinsamer Run

Das muss jemand sein — kein Tool.

Was dieser Flow ist — und was nicht

Prinzipien, kein Pflichtprogramm.

Auf welcher Granularität ihr übergebt,

wie schnell ihr reviewt,

wann ein Artefakt "geteilt genug" ist —

das müsst ihr in eurem Team aushandeln.

Es gibt keine universelle Antwort. Nur die Antwort die zu eurer Kultur, eurer Größe, eurem Tempo passt.

Heute anfangen

1. Repo aufsetzen — einer erstellt, alle klonen

2. context/ gemeinsam befüllen — die erste Konversation die zählt

3. Eine Spielregel festlegen: wie läuft ein Skill-PR ab?

4. Ersten gemeinsamen Flow durchlaufen

5. Learning eintragen — bewusst, als letzten Schritt

Danach: Guardian des Wohnzimmers bestimmen. Dann läuft es.
