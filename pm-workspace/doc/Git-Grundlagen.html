Warum Git?

Jede Änderung ist gespeichert — wer, was, wann, warum.

Kein Überschreiben. Kein Verlust. Jederzeit zurück.

Mehrere arbeiten parallel — ohne Kollisionen.

Änderungen werden reviewt bevor sie für alle gelten.

Im pm-workspace: Skills, context/, CLAUDE.md sind geteilte Infrastruktur.
Git ist der Mechanismus der das kontrolliert ermöglicht.

Die drei Grundbegriffe

Repository (Repo)
Der beobachtete Ordner + seine gesamte Geschichte. Existiert lokal (dein Rechner) und remote (GitHub).

Commit
Ein gespeicherter Zustand — Foto des Repos zu einem Zeitpunkt. Hat Nachricht + Zeitstempel + Autor.

Branch
Eine Abzweigung vom Hauptstand. Du arbeitest isoliert. main bleibt stabil bis du fertig bist.

               \                 /

Branches — warum und wie

Direkt auf main arbeiten = andere blockieren. Branch = dein Zimmer.

# Neuen Branch erstellen und drauf wechseln
git checkout -b skill/decision-brief-risk

# Änderungen machen...

# Stagen und committen
git add .claude/skills/decision-brief/SKILL.md
git commit -m "skill: Risk-Sektion ergänzt"

# Auf GitHub hochladen
git push -u origin skill/decision-brief-risk

# Nach dem Merge: zurück zu main
git checkout main
git pull

.gitignore — was Git ignoriert

Git beobachtet deinen gesamten Ordner. Aber nicht alles soll ins Repo.

Stell dir vor: du hast 50 Prototypen in output/ gebaut.
Jeder Run, jede App, jede Analyse. Gigabytes. Und das ist dein persönliches Zimmer — kein anderer braucht das.

Die .gitignore-Datei sagt Git: "Diese Ordner und Dateien — vergiss sie. Beobachte sie nicht. Committe sie nie."

Du legst die Datei einmal an. Danach passiert es automatisch — für immer, für alle die das Repo klonen.

Keine Angst: die Dateien verschwinden nicht. Sie sind nur für Git unsichtbar.

.gitignore — konkret im pm-workspace

# pm-workspace/.gitignore

# Output-Artefakte — nie committen, jeder macht seine eigenen Runs
output/**
!output/**/.gitkeep

# Python-Umgebung (groß, lokal)
.venv/
__pycache__/

# macOS-Systemdatei (interessiert niemanden)
.DS_Store

Wo liegt die Datei — wo gilt sie?

Eine .gitignore im Root des Repos gilt für das gesamte Repo.
Du kannst auch in Unterordnern eine eigene .gitignore ablegen — dann gilt sie nur dort.
Im pm-workspace: eine Datei im Root, gilt für alles. Das reicht.

Wie Einträge kaskadieren: Regeln werden der Reihe nach gelesen. Eine spätere Regel kann eine frühere aufheben.

1. output/** → ignoriere alles in output/
2. !output/**/.gitkeep → außer: .gitkeep-Dateien doch tracken
Das ! bedeutet immer: "Ausnahme von der Regel darüber."

Eigene Einträge einfach hinzufügen: notes/ · *.local.md

Cheat Sheet — der PM-Workflow

TÄGLICH
git pull                         # Aktuellen Stand holen
git status                       # Wo stehe ich?

ÄNDERUNG MACHEN
git checkout -b typ/beschreibung # Neuen Branch
git add <datei>                  # Stagen
git commit -m "Warum, nicht was" # Committen
git push -u origin <branch>     # PR öffnen auf GitHub

NACH DEM MERGE
git checkout main && git pull    # Zurück + Stand holen

ORIENTIERUNG
git log --oneline                # Was ist passiert?
git diff                         # Was hat sich geändert?

NOTFALL
git stash / git stash pop        # Änderungen kurz weglegen

pull → branch → ändern → add → commit → push → PR → review → merge → pull

Das Wesentliche

Git ist kein Entwickler-Tool.

Es ist Infrastruktur für jeden der kontrolliert zusammenarbeiten will.

Im pm-workspace verbindet Git Zimmer und Wohnzimmer.
